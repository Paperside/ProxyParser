# ProxyParser Product Redesign Design

日期：2026-05-14

状态：已获得方向确认。V1 定位为“半公开模板生态”：个人生成订阅是核心，模板和规则方案可脱敏共享，同时允许把个人扩展订阅共享给指定用户或所有人。

## 1. 产品定位

ProxyParser V1 是一个 Mihomo 扩展订阅工作台。用户导入外部订阅，系统保存快照、提取节点、自动分组、接入规则源，并把用户的分组/规则/配置操作保存成可重放的操作流。最终产物是可被客户端拉取的扩展订阅。

技术栈保留现状：

- 后端：Bun + Elysia + SQLite。
- 前端：React + Vite + TypeScript。
- 数据库迁移：仓库内 SQL migration。
- UI：保留 React 组件体系，但按新产品重新组织页面、导航、信息密度和交互。

现有代码只能作为底座参考，不把现有产品形态当成答案。

## 2. 成功标准

V1 完成时必须满足：

- 用户可以在一个流程内粘贴外部订阅 URL、同步成功、看到节点/分组/规则概览。
- 用户可以生成扩展订阅：系统提取全部节点，生成 `Proxies` 大组和国家/地区组，再应用规则源和用户编辑操作。
- GitHub/HTTP 规则源会定时同步到本地目录，渲染时输出有效 `rule-providers` 和对应 `RULE-SET`。
- 用户可以把规则、分组和配置操作保存为模板，并从源绑定模板中导出不含真实节点的脱敏模板。
- 模板可私有、非公开链接共享、公开共享，扩展订阅可共享给指定用户或所有人。
- 每个扩展订阅有可复制的订阅链接，支持长期 key、短期 key、短期过期时间配置、重置长期 key、立即撤销短期 key。
- 拉取订阅时基于最新可用上游快照重放操作流；失败时落库错误并在可用时返回最近一次成功快照。
- UI 是完整可用的产品界面，不是 API 表单集合；核心操作有清晰反馈、状态、错误、预览和复制结果。
- `bun run typecheck`、后端集成测试、关键前端浏览器检查都通过。

## 3. 核心概念

### 外部订阅

用户导入的 Clash/Mihomo 订阅 URL。系统保存：

- URL、名称、启用状态、可见性。
- 最新同步状态、失败原因、上次成功快照。
- 原始 YAML、解析后的标准化文档、响应头、用量信息。

创建或更新 URL 后应立即尝试同步。同步失败时保留记录，但 UI 必须明确标注不可用于生成。

### 规则源

来自 GitHub raw、HTTP 文件或用户导入 URL 的规则集合。规则源一等化保存：

- `sourceUrl`、`sourceRepo`、branch/path 元数据。
- `behavior`: `domain`、`ipcidr`、`classical`。
- `format`: V1 输出统一为 YAML payload。
- 同步间隔、ETag、Last-Modified、最新缓存、hash、错误。
- 推荐目标代理组和默认插入位置。

### 操作流

扩展订阅保存“用户做了什么”，而不是只保存最终 YAML：

- 节点操作：新增、删除、替换、重命名、启用/禁用。
- 自动分组策略：地区正则、未分类处理、组类型。
- 代理组操作：新增、替换、排序、引用节点/组/provider。
- 规则操作：添加 rule-provider、生成 RULE-SET、插入位置、删除、排序。
- 配置操作：patch 或 full override 顶层配置。

### 模板

模板是操作流的可复用蓝图。模板不应默认包含真实节点。

模板状态：

- `shareable`: 不依赖某个外部订阅的私有对象。
- `source_locked`: 依赖源订阅的原有节点、分组、规则或 provider 名。
- `sanitized`: 从 source locked 模板剥离真实节点后得到的共享版本。

### 扩展订阅

扩展订阅是客户端实际拉取的产物：

```text
外部订阅 + 操作流/模板 + 发布设置 = 扩展订阅
```

内部可沿用 `managed_subscription` 命名，但 UI 统一叫“扩展订阅”。

## 4. V1 工作流

### 4.1 创建扩展订阅

1. 用户点击“新建扩展订阅”。
2. 输入名称并粘贴外部订阅 URL，或选择已有外部订阅。
3. 系统立即同步外部订阅并显示：
   - 节点数量。
   - 源代理组数量。
   - 源规则数量。
   - 用量与过期时间。
   - 解析错误或同步失败原因。
4. 用户进入节点分流步骤。
5. 系统生成 `Proxies` 大组和默认国家/地区组。
6. 用户在规则步骤选择内置或自定义规则源，并为每个规则源指定目标组。
7. 用户在代理组步骤微调组结构。
8. 用户在设置步骤调整 DNS、端口、TUN、sniffer 等配置。
9. 用户预览最终 YAML、校验错误、发布扩展订阅。
10. 系统生成订阅链接，用户复制长期或短期链接。

### 4.2 管理扩展订阅

扩展订阅列表展示：

- 名称、来源、最近同步、最近渲染、节点/规则/组统计。
- 共享状态：私有、指定用户、所有人、非公开链接。
- key 状态：长期 key 是否可用、短期 key 数量、最近拉取。
- 操作：编辑、立即同步并渲染、复制链接、共享、查看快照、停用。

### 4.3 模板沉淀与共享

在扩展订阅编辑器中，用户可以：

- 保存当前操作流为模板。
- 如果模板 source locked，系统解释锁定原因。
- 一键生成脱敏模板：移除真实节点，仅保留分组结构、规则源选择、规则插入顺序、配置操作和自动分组策略。
- 发布到公开市场、非公开链接，或定向共享给用户。

### 4.4 共享个人订阅

扩展订阅共享不是泄露真实 key。共享模型：

- 私有：仅所有者可见和拉取。
- 指定用户：被分享用户可查看扩展订阅配置摘要，并按自己的访问权限生成拉取链接。
- 所有人：可公开查看摘要；是否允许 fork 由 `shareMode` 决定。
- 非公开链接：用于临时展示或发给知道链接的人。

真实节点是否可见由共享模式控制：

- `view`: 可查看配置摘要和最终 YAML 预览，敏感字段默认脱敏。
- `fork`: 可复制模板/操作流，默认不复制真实节点和私有 key。
- `subscribe`: 可拉取扩展订阅，但必须使用有效 key。

## 5. 渲染设计

渲染输入：

- 最新成功外部订阅快照。
- 草稿或模板操作流。
- 规则源目录。
- 发布设置。

渲染输出：

- 最终 Mihomo YAML。
- 结构化文档 JSON。
- 校验结果。
- locked/shareable 判定和原因。
- 统计信息。

关键规则：

- 发布和拉取默认使用最新成功上游快照，不应被草稿上一次预览快照锁死。
- 用户手动选择历史快照预览时，必须只影响预览，不影响正式拉取。
- 渲染失败必须落库 `lastRenderStatus = failed` 和 `lastErrorMessage`。
- 拉取失败时，如果存在最近成功快照，返回降级内容并记录 pull log。

校验必须覆盖：

- 节点名、代理组名、provider 名称唯一。
- 代理组引用的节点/组/provider 存在。
- `RULE-SET` 引用的 rule-provider 存在。
- rule-provider `behavior` 与规则内容匹配。
- `MATCH` 兜底存在且位置合理。
- 自动分组存在未分类节点时给出提示，不直接阻止发布。

## 6. 自动分组

默认分组：

- `Proxies`: 包含所有节点、地区组和 `DIRECT`/`REJECT`。
- `HK`
- `JP`
- `US`
- `TW`
- `SG`
- `KR`
- `Others`
- 可选 `Auto`: `url-test`，用于自动选择低延迟节点。

V1 先使用静态节点引用生成稳定 YAML。后续再增加 Mihomo 原生 `include-all/filter/exclude-filter` 高级模式。

用户可调整：

- 某个节点属于哪个地区。
- 地区组名称和类型。
- 未分类节点放入 `Others`、忽略或手动指定。
- 默认组和规则目标组。

## 7. 规则源与 RULE-SET

选择规则源时，用户必须选择目标策略组。系统保存规则操作：

```json
{
  "type": "attach-rule-provider",
  "providerSlug": "metacubex-geosite-openai",
  "targetPolicy": "AI",
  "insert": { "position": "before-match" }
}
```

渲染时输出：

```yaml
rule-providers:
  metacubex-geosite-openai:
    type: http
    behavior: domain
    url: http://<server>/api/marketplace/rulesets/metacubex-geosite-openai/content
    path: ./rule-providers/metacubex-geosite-openai.yaml
    interval: 86400
    format: yaml
rules:
  - RULE-SET,metacubex-geosite-openai,AI
  - MATCH,Proxies
```

优先使用本服务的缓存内容 endpoint。若部署者配置为直连远端，也可输出远端 URL。

## 8. Key 与订阅链接

长期 key：

- 每个用户已有长期 secret；V1 UI 中应围绕扩展订阅展示长期链接。
- 重置长期 key 后，旧长期链接立即失效。

短期 key：

- 默认 24 小时。
- V1 允许配置 1 小时到 30 天。
- 支持立即撤销某个短期 key。
- UI 展示过期时间、最近使用时间、状态。

拉取 URL：

```text
GET /subscribe/:subscriptionId?token=<key>
```

## 9. UI 设计方向

视觉方向：高密度网络控制台。冷静、精确、可扫描，适合反复操作。

导航：

- 工作台
- 扩展订阅
- 外部订阅
- 规则源
- 模板
- 共享
- 设置

页面原则：

- 首页直接展示工作台和最近扩展订阅，不做营销页面。
- 扩展订阅编辑器是主体验，采用“左侧步骤 + 中间编辑 + 右侧诊断/预览”的布局。
- 规则和分组操作使用列表、分段控件、菜单、图标按钮和明确状态标签。
- 所有保存、同步、渲染、复制、撤销操作都有反馈。
- 文案中文；API 字段和数据结构英文。
- 不使用大面积装饰性渐变和卡片堆叠，避免页面像模板站。

## 10. 实现边界

首版产品闭环必须实现：

- 外部订阅导入并立即同步。
- 扩展订阅向导可从 URL 直接创建来源。
- 自动节点分流接入新渲染链路。
- 规则源渲染为有效 `rule-providers` 和 `RULE-SET`。
- 操作模板保存、source locked 判定、脱敏模板导出。
- 扩展订阅共享字段与基本 UI：私有、指定用户、所有人。
- 长期/短期 key UI，短期 key TTL 限制和撤销。
- 渲染失败落库。
- 新 UI 可通过浏览器完成端到端主流程。

延期：

- 复杂团队/组织权限。
- 真实 Mihomo core 校验器。
- 结构化 YAML diff。
- 多客户端兼容矩阵。
- Mihomo `include-all/filter` 高级自动分组模式。

## 11. 测试策略

后端集成测试覆盖：

- 创建外部订阅并立即同步成功。
- 拉取时使用最新快照重放操作流。
- 自动分组生成 `Proxies` 和地区组。
- 规则源引用生成 `rule-providers` 与 `RULE-SET`。
- 渲染失败落库。
- 短期 token TTL 限制和撤销。
- 脱敏模板不包含真实节点。

前端验证覆盖：

- TypeScript typecheck。
- 浏览器打开本地应用，完成登录、创建来源、创建扩展订阅、预览、发布、复制链接。
- 检查核心页面在桌面宽度无明显重叠、文本溢出、空白主体验。
