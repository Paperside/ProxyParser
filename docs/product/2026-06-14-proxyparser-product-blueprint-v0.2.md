# ProxyParser 产品蓝图 v0.2

日期：2026-06-14  
状态：产品方案草案，可持续迭代  
对应交互原型：`designs/proxyparser-product-console/index.html`

## 0. 本次修订范围

本版本基于 v0.1 继续补充，重点解决：界面不展示讨论性文案、侧边导航分级、构建器只从新增/编辑进入、访问与共享集成到扩展订阅详情、长期 Key 放入设置、多来源合并强制 Full Override、节点/代理组/规则之间通过稳定 ID 联动、规则源导入为规则条目而非默认长期绑定、模板只面向 Full Override 并记录可重放操作。

## 1. 用户界面原则

产品界面面向用户，不面向产品讨论。界面应展示页面名称、对象状态、可执行动作、操作后果和下一步，而不是“产品愿景式”标题。

例如：

- 页面标题用“工作台”，不要用“把所有订阅变成一条稳定长期链接”。
- 工作台可以有“产品心智”板块，用用户语言说明：外部订阅是原料、扩展订阅是成品、模板是构建逻辑。
- 批量清洗、规则去重、模板提炼等可能改写用户输入的操作，必须先展示预览和确认。

## 2. 信息架构

侧边导航：

```text
工作台
订阅
  外部订阅
  扩展订阅
规则源
模板
  我的模板
  模板共享
设置
```

不再设置一级“构建器”Tab；构建器只能从新增或编辑扩展订阅进入。  
不再设置一级“访问与共享”Tab；访问、共享、短期链接、拉取日志进入扩展订阅详情。  
长期 Key 是账号级能力，放入设置页；扩展订阅详情只提供“复制长期链接”。

## 3. 页面主按钮规则

外部订阅页：页面级主按钮只有“添加外部订阅”。“基于此创建扩展订阅”放在对应外部订阅卡片或详情页中。

扩展订阅页：页面级主按钮只有“新增扩展订阅”。“复制长期链接”“编辑构建逻辑”“短期链接”“共享”等对象级操作放在对应扩展订阅卡片或详情页中，避免顶栏重复。

模板页：父级为“模板”，子级为“我的模板”和“模板共享”。模板共享承载模板市场、共享模板、规则包共享，不单独做“市场”一级入口。

## 4. 构建器入口与来源规则

构建器流程：

```text
来源 -> 模式 -> 节点 -> 代理组 -> 规则 -> 配置 -> 预览 -> 发布
```

来源可以选择单个外部订阅，也可以选择多个外部订阅合并。

多来源合并必须强制 Full Override，且只能合并节点：

- 读取每个来源的节点。
- 标记节点来源。
- 不合并源代理组、源规则、源 rule-provider、源顶层配置。
- 后续由用户重新生成或手动创建代理组和规则。

原因：多个来源的规则、代理组、rule-provider 和配置冲突风险太高，自动合并会制造不合法配置。

## 5. 模式规则

### Patch / 在源配置基础上修改

保留源订阅原有节点、代理组、规则和配置，只记录用户对当前订阅的局部修改。

适合：上游配置本身可用，用户只做小修小补。  
不适合：多来源合并、模板共享、完整重组自己的规则和代理组。

### Full Override / 完整重组

只消费节点，重新组织代理组、规则和配置。

适合：用户想维护自己的分流逻辑、应用模板、沉淀模板、合并多个订阅节点。  
这是 ProxyParser 的主线模式。

只有 Full Override 产生的构建逻辑才进入可共享模板主路径。Patch 默认是 source-locked 私有修改。

## 6. 节点模型

节点名称不能作为内部引用主键。ProxyParser 必须为节点分配稳定内部 ID。

节点字段建议：

```text
nodeId
sourceType: native | custom
sourceId
sourceNodeFingerprint
displayName
protocol
server
port
tags
regionHint
enabled
rawConfig
```

`native` 表示来自外部订阅的原生节点。  
`custom` 表示用户在 ProxyParser 中新增的自定义节点，例如自己的 VPS。

节点页默认只列出当前可用节点，不自动重命名、不自动清洗、不自动分地区。用户可主动触发：去除 emoji、去除倍率标签、正则替换、批量禁用、批量删除。所有批量改写必须先展示变更预览。

节点页核心功能：查看节点详情、编辑节点字段、新增自定义节点、删除节点、启用/禁用节点、标记来源、打标签。

地区分类不属于节点页默认主操作。地区分类属于代理组步骤，由用户点击“生成地区代理组”。

当节点重命名时，内部引用不变，代理组和规则最终渲染名自动更新。  
当节点删除时，所有代理组成员引用自动移除；若规则目标引用该节点，必须要求用户重新选择目标或删除规则。

## 7. 策略目标 Policy Target

产品中统一把代理组、可被规则指向的自定义节点、内置目标叫“策略目标”。

策略目标包括：

- 已知代理组。
- ProxyParser 自定义节点。
- 内置目标：`DIRECT`、`REJECT`、`PASS`、`COMPATIBLE` 等。

代理组成员选择器和规则目标选择器都只允许选择已知策略目标，避免用户手写不存在的名称。

## 8. 代理组模型

代理组成员只能来自：已知节点、已知代理组、内置策略目标。高级 RAW 模式可以保留，但保存前必须结构化解析和校验。

代理组步骤提供：

- 生成 `Proxies` 总代理组。
- 生成国家/地区代理组。
- 一键生成推荐服务代理组，例如 AI、Streaming、Telegram、Ads、Apple、Microsoft。
- 清空代理组。
- 新增自定义代理组。

Full Override 模式下可提供“一键推荐代理组”；Patch 模式下不自动生成，除非用户主动选择。

ProxyParser 保存代理组操作，而不是只保存最终结果。例如：

```text
create_group Proxies with preset all-nodes
create_region_groups with region_rules default
create_service_group AI with member_preset default-proxy
create_service_group Ads with member_preset reject-first
insert_custom_node vps-sg-01 into group AI at top
remove_group Netflix
rename_group Proxy -> Proxies
```

标准节点配置 Preset：

- 全部节点配置：包含所有启用节点。
- 地区配置：按地区计算节点。
- 默认代理配置：`Proxies`、地区组、`DIRECT`。
- 默认拒绝配置：`REJECT` 放前面，适合广告拦截。
- 默认直连配置：`DIRECT` 放前面，适合国内服务。
- 自定义配置：用户手动选择成员。

模板中优先记录 preset 和命令，不记录一次性计算后的原生节点列表。

## 9. 原生节点与自定义节点的模板规则

模板可以稳定记录：创建代理组、标准节点配置、引用已知代理组、引用内置目标、引用 ProxyParser 自定义节点、将自定义节点插入某个代理组顶部或底部。

模板不能稳定记录：直接引用某个外部订阅原生节点、依赖某个外部订阅独有节点名称。

如果用户在代理组或规则中手动加入了原生节点，提炼模板时必须提示：

```text
该成员来自外部订阅原生节点，无法作为通用模板的一部分保存。
将从模板中移除该成员，或仅保存为空位。
```

最坏情况下，只记录创建了该代理组，但成员为空，并在提炼报告中说明原因。

## 10. 规则模型

规则配置的目标应该是代理组、自定义节点或内置策略目标，而不是让用户自由输入任意字符串。

默认不允许直接选择外部订阅原生节点作为规则目标。若高级模式允许，模板提炼时应判定为 source-locked 或从模板中剔除。

规则源的默认语义是“导入规则”，不是“长期绑定规则源”。流程：

1. 拉取规则源内容。
2. 解析规则条目。
3. 用户选择目标策略目标。
4. 插入到当前规则链指定位置。
5. 保存为当前扩展订阅自己的规则操作。

导入后，规则条目属于当前扩展订阅构建逻辑，不再隐式绑定原规则源。可以提供“重新从规则源刷新导入”的显式按钮，但刷新前必须展示新增、删除、去重、目标失效等变更。

规则步骤支持：从规则源导入规则、从规则包导入规则、粘贴多条规则、添加单条规则、删除规则、调整插入位置、添加前置规则、添加后置规则、添加 MATCH、去重、合法性校验。

规则类型固定化：DOMAIN、DOMAIN-SUFFIX、DOMAIN-KEYWORD、DOMAIN-WILDCARD、DOMAIN-REGEX、GEOSITE、IP-CIDR、IP-CIDR6、GEOIP、PROCESS-NAME、PROCESS-PATH、DST-PORT、RULE-SET、MATCH、AND、OR、NOT。

粘贴多条规则时，系统必须展示：成功解析数、去重数、非法数、目标不存在数、规范化结果。如果最终保存内容与用户原始输入不完全一致，必须让用户确认。

## 11. 模板模型

可共享模板只来自 Full Override / 完整重组模式。

模板需要记录：

- ProxyParser 自定义节点。
- 节点标签与自定义节点插入位置。
- 代理组创建命令。
- 标准节点配置。
- 规则导入操作和手写规则。
- 规则目标选择。
- 配置覆盖。

模板必须区分原生节点和自定义节点。自定义节点可以进入模板；原生节点不进入通用模板。

提炼模板时必须展示报告：

```text
将保存：
- 自定义节点 2 个。
- 代理组创建命令 9 条。
- 标准节点配置 5 个。
- 规则导入与手写规则 42 条。
- DNS / TUN / Sniffer 配置。

不会保存：
- 直接引用的原生节点 JP-01。
- 针对源订阅原生代理组 Proxy 的修改。
- 源订阅专属 rule-provider 引用。
```

应用模板到外部订阅时，系统导入模板中的自定义节点，根据当前外部订阅重新计算地区组，重新执行代理组命令，重新插入规则，校验策略目标是否存在，再生成预览。

## 12. 访问与 Key

访问与共享属于扩展订阅详情：分享给指定用户、非公开访问链接、短期订阅链接、短期链接列表、撤销短期链接、拉取日志。

长期 Key 属于设置页：当前长期 Key 状态、最近使用时间、重置长期 Key、默认订阅链接域名、User-Agent 设置。

扩展订阅详情页可以提供“复制长期链接”，但长期 Key 的重置和默认设置不放在扩展订阅详情中。

## 13. UI 组件规范

正式前端应使用统一 UI 组件库或自建组件系统，避免浏览器原生 select、prompt、confirm 混入正式界面。

关键组件：Button、Card、Tabs、Dialog、Select/Combobox、Toast、Badge、Data Table、Empty State、Inline Validation、Diff Viewer。

Select/Combobox 组件必须支持：选项标题、选项说明、状态徽标、禁用原因、搜索。

例如共享模式选项应展示：

```text
不共享
仅自己可见，不允许他人查看、fork 或拉取。

允许 fork
对方可复制脱敏构建逻辑，不能获得真实节点或 Key。

允许 subscribe
授权用户可拉取这个扩展订阅。
```

不要出现原生浏览器下拉样式，也不要出现 `view + fork disabled` 这种面向内部状态的文案。

## 14. 仍需讨论的问题

1. 规则源导入后是否完全脱钩，还是支持“静态导入”和“保持 rule-provider 引用”两种模式。
2. 自定义节点包含密码、UUID、private-key 等敏感字段，私有模板和公开模板的存储、脱敏、加密策略要明确。
3. 原生节点如何稳定 fingerprint：sourceId、server、port、protocol、凭据 hash、name fallback 等。
4. 大型规则源全部展开为内联规则可能导致 YAML 体积过大，需要评估 rule-provider 输出策略。
5. RAW 模式是否允许保存未知目标，若允许，应如何标记 source-locked 与不可共享。
