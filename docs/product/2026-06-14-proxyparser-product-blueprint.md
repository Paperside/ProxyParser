# ProxyParser 产品蓝图 v0.1

日期：2026-06-14  
状态：产品方案草案，可持续迭代  
对应原型：`designs/proxyparser-product-console/index.html`

## 0. 一句话定位

ProxyParser 是一个面向 Clash / Mihomo 用户的订阅配置控制台：它托管外部订阅，抽取节点，重组代理组与规则，把组装逻辑沉淀成可复用模板，并稳定发布为可长期拉取、可共享、可审计的扩展订阅。

用户以后不直接订阅机场链接，而是订阅 ProxyParser 给出的长期链接。机场订阅只是原料，ProxyParser 才是用户真正维护的配置中心。

## 1. 产品北极星

ProxyParser 解决的不是“订阅格式转换”，而是“外部订阅会变，但我的配置逻辑不该反复重做”。

核心承诺：

1. 托管外部订阅。
2. 看懂外部订阅。
3. 重组外部订阅。
4. 记住重组逻辑。
5. 把逻辑复用成模板。
6. 把最终结果稳定发布。
7. 把模板、规则源、扩展订阅安全分享给别人。

最终用户心智：

```text
我把机场订阅交给 ProxyParser。
ProxyParser 帮我看懂它。
我在 ProxyParser 里搭自己的分流方案。
ProxyParser 记住这个方案。
以后机场节点怎么变，我的方案都继续生效。
我只需要复制 ProxyParser 的长期链接给客户端。
```

## 2. 核心对象模型

### 2.1 外部订阅 External Subscription

外部订阅是原料。它来自机场、订阅服务、GitHub raw、HTTP YAML，或者未来的多个来源合并。

外部订阅负责：

- 保存原始 URL。
- 定时同步。
- 保存历史快照。
- 解析 YAML。
- 提取 `proxies`、`proxy-providers`、`proxy-groups`、`rule-providers`、`rules`、顶层配置。
- 提取订阅用量和过期时间。
- 记录同步错误。
- 保留原始内容。

外部订阅不是用户最终给客户端订阅的对象。它是上游数据源。

### 2.2 快照 Snapshot

每次成功同步外部订阅，都生成快照。

快照包含：

- 原始 YAML。
- 标准化 JSON。
- 同步时间。
- 响应头。
- 订阅用量。
- 内容 hash。
- ETag / Last-Modified。
- 解析警告。
- 兼容性诊断。

用户可以查看历史快照并比较：新增节点、删除节点、代理组变化、规则变化、`rule-provider` 变化、顶层配置变化。

### 2.3 扩展订阅 Extended Subscription

扩展订阅是用户真正使用的成品。

```text
外部订阅 + Recipe + 发布设置 + 访问控制 = 扩展订阅
```

扩展订阅负责：

- 选择一个或多个外部订阅。
- 消费节点。
- 应用分组逻辑。
- 应用规则源。
- 应用模板。
- 应用配置覆盖。
- 渲染最终 YAML。
- 校验最终 YAML。
- 生成长期链接。
- 生成短期链接。
- 记录拉取日志。
- 失败时降级返回最近成功版本。

### 2.4 Recipe / 组装逻辑

Recipe 是产品最核心的数据结构。ProxyParser 不应该只保存最终 YAML，而应该保存“用户是怎么从外部订阅生成最终配置的”。

Recipe 记录：

- 节点处理：使用全部节点、排除节点、节点重命名、地区识别、手动地区修正。
- 代理组处理：自动生成地区组、生成 Auto、新增策略组、修改/删除已有组、完整覆盖源代理组。
- 规则处理：引入 `rule-provider`、指定目标策略组、指定插入位置、自定义规则、删除源规则、完整覆盖规则链。
- 配置处理：保留源配置、patch 源配置、full override 顶层配置、DNS/TUN/sniffer/mixed-port 等设置。
- 发布处理：长期链接、短期链接、可见性、分享权限。

### 2.5 模板 Template

模板是可复用 Recipe。模板默认不包含真实节点。

模板状态：

- 通用模板：不依赖某个外部订阅，只消费抽象节点集合，可应用到任何外部订阅。
- 源绑定模板：依赖某个外部订阅的具体代理组、规则、provider 名称，只能安全应用在这个源或相似源上。
- 脱敏模板：从源绑定模板中剥离真实节点和敏感字段后得到的可分享版本。

### 2.6 规则源 Rule Source

规则源是 `rule-provider` 的来源。它可以来自：

- blackmatrix7/ios_rule_script。
- MetaCubeX/meta-rules-dat。
- Loyalsoldier/clash-rules。
- ACL4SSR。
- GitHub raw URL。
- HTTP 文件。
- 用户自定义 inline payload。

规则源应该有独立生命周期：导入、识别 behavior、同步、缓存、预览内容、校验、被扩展订阅/模板/规则包引用。

### 2.7 规则包 Rule Pack

规则包是一组规则源与目标策略组的组合。

例如：

- AI 规则包：OpenAI、Anthropic、Gemini、Perplexity → AI。
- 流媒体规则包：Netflix、Disney、YouTube、Spotify → Streaming。
- 中国直连规则包：CN domain、CN CIDR、Private → DIRECT。

规则包可以被模板引用，也可以单独分享到市场。

### 2.8 市场 Marketplace

市场不公开真实订阅节点。市场分享的是：模板、规则源、规则包、配置片段、扩展订阅摘要。

### 2.9 访问控制 Access Control

访问控制负责长期订阅链接、短期订阅链接、指定用户分享、非公开链接、公开查看、允许 fork、允许 subscribe、撤销访问、重置长期 Key、查看拉取日志。

产品原则：长期链接是主路径，短期链接是辅助能力。

## 4. 核心页面

### 4.1 工作台

工作台是控制台首页，不是营销页。它应该展示：

- 今日状态：外部订阅数、扩展订阅数、同步失败、渲染失败、快过期订阅、拉取次数。
- 常用操作：新建扩展订阅、添加外部订阅、导入规则源、浏览模板市场、复制最近使用的长期订阅链接。
- 最近扩展订阅：名称、来源、节点数、规则数、最近生成、最近拉取、复制长期链接。
- 异常中心：同步失败、规则源失败、渲染失败、引用不存在、MATCH 缺失、behavior 不匹配。

### 4.2 外部订阅

外部订阅列表展示名称、URL、同步状态、最近同步、节点数、proxy-provider 数、代理组数、rule-provider 数、规则数、用量、过期时间、被多少扩展订阅使用。

外部订阅详情包含：概览、节点、proxy-providers、代理组、rule-providers、规则、顶层配置、原始 YAML、快照历史、使用情况。

### 4.3 扩展订阅

扩展订阅列表展示名称、来源、模板、生成状态、节点数、代理组数、规则数、最近生成、最近拉取、可见性。

主按钮必须是“复制长期链接”。其他操作：编辑、立即生成、查看详情、短期链接、共享、历史、停用、删除。

### 4.4 扩展订阅构建器

构建器是产品主流程，结构为：来源、构建模式、节点、代理组、规则、配置、预览、发布。

构建模式：

1. 在源配置基础上修改：保留源订阅原有代理组、规则、rule-provider 和配置，只记录修改。容易 source_locked。
2. 只消费节点，完整重组配置：忽略源订阅原有代理组和规则，只使用节点，重新生成代理组、规则和配置。更适合模板化。
3. 应用模板：选择已有模板，把当前外部订阅的节点灌入模板。

### 4.5 规则源

规则源中心展示内置规则库、我的规则源、规则源详情、导入 GitHub 规则库、规则包。

每个规则源展示名称、来源库、behavior、规则数量、最近同步、状态、被引用次数。

### 4.6 模板

模板模块负责把一套可复用的组装逻辑保存下来，应用到不同外部订阅。

模板详情包含概览、Recipe、代理组结构、规则源、配置、版本、使用情况、共享。

### 4.7 市场

市场包含模板市场、规则包市场、规则源市场、扩展订阅分享。默认不泄露真实节点。

### 4.8 访问与共享

访问页集中展示长期 Key、短期 Key、分享授权、非公开链接、公开访问、拉取日志。

### 4.9 设置

设置放账号级能力：账号信息、默认 User-Agent、同步间隔、长期 Key 管理、安全设置、API base URL、反代设置、数据目录、备份恢复、审计日志。

## 5. 状态设计

### 5.1 外部订阅状态

- 未同步。
- 同步中。
- 同步成功。
- 同步失败。
- 已过期 / 可能失效。
- 已停用。

### 5.2 解析状态

- 解析成功。
- 部分兼容。
- 解析失败。

部分兼容意味着可以提取核心数据，但存在缺失或警告，例如缺少 proxy-groups、部分规则无法识别、proxy-provider 暂不能展开。

### 5.3 扩展订阅状态

- 草稿。
- 待预览。
- 预览成功。
- 预览失败。
- 已发布。
- 生成成功。
- 生成失败。
- 降级可用。
- 已停用。

“降级可用”表示本次生成失败，但仍可返回上一次成功 YAML。

### 5.4 模板状态

- 草稿。
- 可分享。
- 源绑定。
- 已脱敏。
- 已发布。
- 已归档。

### 5.5 规则源状态

- 未同步。
- 同步成功。
- 同步失败。
- 缓存可用。
- 缓存过期。
- 格式错误。
- behavior 不匹配。

### 5.6 分享状态

- 私有。
- 指定用户。
- 非公开链接。
- 公开。
- 已撤销。
- 已过期。

## 6. 模板化判定规则

### 6.1 可成为通用模板

满足：

- 不包含真实节点。
- 不依赖源订阅 URL。
- 不引用源订阅特定 proxy-provider。
- 不引用源订阅特定 rule-provider。
- 不 patch 源订阅已有规则。
- 不 patch 源订阅已有代理组。
- 使用通用规则源。
- 使用抽象节点集合。

### 6.2 会 source_locked

出现：

- 修改源代理组。
- 删除源代理组。
- 修改源规则。
- 删除源规则。
- 修改源 rule-provider。
- 引用源 provider 名称。
- 引用源特定组名。
- 修改源顶层配置。
- 修改源节点。

## 7. 核心用户旅程

### 7.1 第一次使用

1. 添加外部订阅。
2. 系统同步并解析。
3. 用户基于外部订阅创建扩展订阅。
4. 选择“只消费节点，重新组装配置”。
5. 节点自动分组。
6. 创建 AI、Streaming、Telegram 等策略组。
7. 添加 OpenAI、Netflix、Telegram、CN、Reject 等规则源。
8. 配置 DNS/TUN/Sniffer。
9. 预览并校验。
10. 发布并复制长期订阅链接。

### 7.2 上游订阅更新

1. 外部订阅同步到新快照。
2. 扩展订阅基于最新成功快照重放 Recipe。
3. 新节点自动进入地区组和 Proxies。
4. 规则和配置保持不变。
5. 长期链接不变。

### 7.3 沉淀模板

1. 用户在扩展订阅详情点击“保存为模板”。
2. 系统判断是否 shareable。
3. 默认创建脱敏、通用模板。
4. 用户可应用到其他外部订阅。

### 7.4 分享模板

1. 用户发布模板到市场。
2. 系统检查不包含真实节点和私有 Key。
3. 其他用户应用模板到自己的外部订阅。

### 7.5 临时分享订阅

1. 用户创建 24 小时短期链接。
2. 可标记用途，例如“发给朋友”。
3. 可以随时撤销。
